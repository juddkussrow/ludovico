Five frontier multi-agent frameworks proposed: (1) Delegation — parent spawns sub-agent, receives response; (2) Creator-Verifier — separate agents for build/verify; (3) Direct communication — agents communicate without central coordinator (state fragmentation risk); (4) Negotiation — agents compete/cooperate over shared resources (net positive-sum preferred); (5) Broadcast — one agent informs many (status, context, constraints). "Missions" architecture combines four (delegation, creator-verifier, broadcast, negotiation) into sustained workflow. Three-role system: Orchestrator (planning, strategic questions, validation contracts defining success before coding), Workers (clean context, reads spec, implements feature, commits via Git for clean slate handoff), Validators (scrutiny validator runs tests/type-check/lints/spawns code-review agents; user-testing validator acts as QA via computer-use, clicks forms, tests flows holistically). Validation contracts (created during planning) define correctness independently of implementation—hundreds of assertions per complex project. Structured handoffs between agents detail: what completed, what remains, commands run and exit codes, issues discovered, adherence to procedures. Longest mission ran 16 days (full sprint length); believed runnable to 30 days via structure. Serial execution (not parallel): features run sequentially; one worker/validator at a time; parallelization allowed only on read-only ops (codebase search, API research, code review). Error rate drops dramatically vs. parallelism; coordination overhead avoided. Mission Control UI tracks: active worker, handoff summary, validator findings, budget burn-down, asynchronous monitoring. Model selection per role critical ("droid whispering"): planning uses slow reasoning, implementation uses fast code fluency, validation uses precise instruction-following. No single model/provider excels at all three. Production data on Slack-clone build: 60% time on implementation, 60% tokens on implementation; validation never succeeds first pass (follow-up features always created); 50% of final LOC is tests; 90% code coverage. Validation contracts + milestone checkpoints allow missions to run successfully even with open-weight models. System design encourages continuous improvement: orchestration logic in prompts/skills (700 lines + 4-sentence tweaks shift execution strategy), worker behavior driven by per-mission skills, minimal hard-coded logic lets models do their best while system handles bookkeeping (validation, handoff enforcement).
