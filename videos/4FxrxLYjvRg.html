Problem: one slow dependency (e.g., Service B) consumes shared thread pool while healthy services (C, D) starve. Retries worsen the collapse by adding load to drowning service. Solution: Circuit Breaker (three states: closed—normal operation; open—fail fast on error threshold exceeded, no wait-time; half-open—one test request after cooldown to probe recovery). Half-open is the recovery probe: single test request avoids flooding recovering service. Bulkhead pattern: isolate resources per dependency. Shared 100-thread pool → per-service pools (e.g., 20 threads each for B, C, D). When B fails, only its 20 threads are consumed; beyond allocation, requests rejected immediately. Circuit Breaker decides *when* to stop; Bulkhead decides *how much* of the system a failure can consume. Together: Bulkhead contains blast radius from moment failure starts; Circuit Breaker stops wasting effort once pattern is clear. Neither alone is sufficient—without Bulkhead, window exists before breaker trips where damage spreads; without Circuit Breaker, test requests waste time on half-open probes. Resilience4j library provides both patterns; configuration is simple (failure threshold, timeout for breaker; concurrent call limits for bulkhead).
