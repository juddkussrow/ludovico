Server-Sent Events (SSE) · no reliable link · other · mentioned SYNOPSIS: WebSockets solve the HTTP request-response limitation for real-time server-to-client pushes. Pre-WebSockets workarounds: polling (ask every seconds) and long polling (hold connection, reconnect on data). WebSockets: one persistent TCP connection, full-duplex (both sides send anytime), no handshake cost per message. Design decisions: (1) Sec-WebSocket-Key and magic GUID (RFC constant string) defeat transparent proxy cache poisoning—client generates random key, server concatenates GUID, hashes with SHA-1, sends back; proxy cache miss if key doesn't match. (2) HTTP upgrade request warms TCP slow start: initial bytes flowing in the upgrade handshake grow the TCP congestion window, so by the time WebSocket frames arrive, the connection is already warm (avoids slow-start penalty twice). (3) Client frame masking: all outgoing frames are XOR'd with a random 4-byte key to prevent cache-poisoning attacks (malicious webpage crafting frames that look like valid HTTP responses to poison shared proxies); servers are trusted, so server→client frames need no masking. (4) Ports 80/443: reuse HTTP/HTTPS ports so corporate firewalls (which block custom ports) allow traffic; deep-packet inspection sees HTTP upgrade, by then safely through. (5) HTTP/2 multiplexing comparison: HTTP/2 can multiplex multiple streams on one TCP connection with server-sent events, avoiding WebSocket's multi-connection overhead; but WebSockets win for true bidirectional unpredictable traffic (gaming, collab editing). Real-world problem-solving evident in each design choice. BEATS: Pre-WebSockets polling burned bandwidth; long polling tore down/rebuilt HTTP constantly; HTTP upgrade handshake serves dual purpose as slow-start warm-up; client masking blocks cache poisoning; port 80/443 penetrates firewalls; HTTP/2 SSE is simpler for server-push use cases ENTITIES: TCP, HTTP, HTTP/2, SHA-1, XOR, RFC 6455, transparent proxies, deep-packet inspection, firewall policy, congestion control VERDICT: WebSockets are elegant solutions to real-world protocol and network problems. Understanding these five design decisions prepares developers for system design interviews and clarifies why WebSockets work reliably across the internet.
