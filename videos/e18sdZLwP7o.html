They live in .claude/agents/ (project-level) or globally. Subagents differ from skills: subagents run in fresh chat sessions with clean context windows (preserving main-session context), can run in parallel (many independent sessions), and support different models (e.g., main session on Opus, subagents on Haiku to save cost). Skills are typically invoked within the same session (no clean context). Core YAML front matter includes: name (agent identifier), description (trigger for progressive disclosure—Claude reads YAML only to decide if agent is relevant, then reads full instructions if match), model, tools, disallowed_tools, color (visual identifier), memory scope (project/user/local/none). Progressive disclosure: Claude Code's default behavior searches through agent descriptions to auto-invoke matching agents without wasting tokens on reading full instructions if irrelevant. Building workflow: (1) Manual configuration or (2) generate with Claude (describe agent purpose and trigger conditions, system outputs markdown file). Example: "Create a sub-agent that criticizes my work, plays devil's advocate, looks for flaws. Invoke on 'roast my plan' or 'review my plan'." System generates YAML + instructions as a new agent file. Key design rules: precise descriptions prevent misfires (agent doesn't fire when expected, or fires unexpectedly); instructions should define task sequencing and skill invocation; test iteratively and refine descriptions based on actual behavior. Use cases: (1) Keep main context clean—delegate heavy reads/research to subagent so main session doesn't accumulate noise; (2) Parallel work—spin up multiple subagents simultaneously on independent tasks (e.g., one for API design, one for testing, one for docs) and collect results back in main session; (3) Specialist expertise—each subagent is an expert (security auditor, TypeScript wizard, GraphQL architect) with tailored instructions and tools; (4) Cost optimization—use Haiku/Sonnet subagents for routine work (summarization, reading, research), reserve Opus for complex reasoning in main session; (5) Fresh perspective—subagents can wake with zero memory or fresh context, enabling unbiased review. When NOT to use: quick edits (no need for subagent overhead), sequential dependent steps, or when subagent needs full conversation context. Awesome Claude Code Subagents GitHub repo: open-source library of community-built agents available for reuse (curated by subject matter experts). Read-only subagents: restrict via disallowed_tools (e.g., deny write/edit) and MCP server restrictions; can verify third-party subagents with a "malware checker" subagent before use. Dynamic workflows (Opus 4.8+): auto-generate many subagents in parallel for large projects (expensive, use cautiously). Session limits: max turns can be set per subagent to prevent infinite loops.
