Wasm Beyond the Browser: Use Cases at Scale - Colin Murphy & Sean Isom, Adobe

This video explores Adobe's current and potential use cases for WebAssembly (Wasm) beyond the browser, specifically in edge compute and data centers. The presenter demonstrates an image processing application rewritten in Rust and deployed across various Wasm platforms, including wasmCloud, Cloudflare, and Fastly. The talk highlights Wasm's advantages in performance, cost, security, and compliance compared to traditional containerization like Docker, while also addressing challenges like single-threaded WASI and integration with existing cloud infrastructure.

Adobe has been using Wasm for years to bring C++ codebases to the web. The presentation focuses on Wasm's potential for edge compute and data centers, driven by needs for improved performance, cost efficiency, security, and compliance, especially in contrast to Docker containers which can lead to numerous CVEs and auditing challenges (e.g., FedRAMP). Murphy demonstrates a proof-of-concept: an image processing application (background removal) rewritten in Rust and deployed across various Wasm environments. This application was successfully run in the browser, on Fastly, Cloudflare, and wasmCloud (within Adobe's Kubernetes cluster). The demo shows the application running live on an Adobe Kubernetes cluster via wasmCloud, and then on Cloudflare, performing background removal on a signature image. The presenter notes that Fastly uses Wasmtime, while Cloudflare uses V8, but the difference was not significant for this trivial application. Another demo showcases Cloudflare Durable Objects, used for collaborative editing, where changes in one browser instance are reflected in another, demonstrating low-latency state persistence at the edge. A third demonstration, developed by Christoph Bruving (BMW) and Steve Schultze (Cosmonaut), highlights WasmCloud's capability for ONNX and TensorFlow inference at the edge, showing an image being processed to generate a content authenticity fingerprint locally. This addresses the challenge of running large ML models that are too big for browsers and suffer from high latency if processed server-side. Key lessons learned include that C++ code for browsers (V8) does not "just work" with WASI, and that while Wasm is more efficient than JVM/Spring Boot, the efficiency argument alone isn't enough to justify a major architectural shift for large organizations with existing Java investments. Challenges remain with single-threaded WASI for compute-heavy applications. Platform requirements for Adobe include reliability at scale, isolation of untrusted code, running in Adobe cloud accounts (AWS, Azure), compatibility with cloud resources and identity (AWS IAM, Azure AD, SPIFFE), and Kubernetes integration. The presenter recommends writing new greenfield projects in Rust and leveraging Wasm's portability.
