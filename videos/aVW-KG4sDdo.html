A frozen student model (e.g., GPT-5.5) executes tasks in a harness (CodeX or Claude Code). An optimizer model (also GPT-5.5, unfrozen) reads execution traces—task metadata, tool calls, API responses, compiler errors, final answers—and deduces systematic failure patterns. The optimizer proposes textual edits (append, replace, delete) to the skill.md file, constrained by an edit budget (top-3 to top-10 edits, decaying schedule over epochs). Fast updates (per-batch) learn from current data; slow updates (epoch-wise) detect systemic drift and reflect on previous+current skills. Validation occurs in sandbox (code execution, real errors). Results on benchmarks (SearchQA, Spreadsheet, Office QA, AlfWorld, HotpotQA): SkillOpt achieves 87.3% (SearchQA), 84% (Spreadsheet), 80% (Office QA) on GPT-5.5 bare model; with harness, improvements vary (Codex: +5.5%, Claude Code: +4%). Learned edits are small (145–883 tokens) but impactful: often 1–3 edits per domain (e.g., "Infer expected answer type from clue wording; choose shortest canonical entity with co-occurring evidence"). Key insight: optimizer has no external contact, only reads text traces—results are harness syntax optimization + procedural knowledge injection. Paper includes bonus study on model-generated skill lifecycle (raw experience → skill extraction → skill consumption). BEATS: No specific video timestamps provided in description.
